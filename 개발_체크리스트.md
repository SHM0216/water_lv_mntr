# 빗물펌프장 모바일 앱 개발계획

**대구광역시 · 빗물펌프장 스마트 관리시스템**
SCADA 연동 · 30초 주기 실시간 수위 모니터링 · 4단계 경보 시스템

- 총 6단계 · 약 12~18주
- 우기(6월) 이전 파일럿 안정화 목표
- 총 34개 작업 항목

---

## 시스템 아키텍처 개요

```
[1. 폐쇄 OT망]
  빗물펌프장 22개소 (PLC · 내수위/외수위 센서)
      →
  SCADA 서버 (OPC UA · Modbus 수집)
      ↓ 단방향 (읽기 전용)

[2. DMZ · 경계구간]
  단방향 게이트웨이 (Data Diode · 읽기만 허용)
  수집 에이전트 (30초 주기 · MQTT/HTTPS)
      ↓ TLS 1.3 + 인증서

[3. 백엔드 서버 (클라우드)]
  FastAPI (REST · WebSocket · 인증)
  TimescaleDB (시계열 저장 · 22개소 × 30초)
  알람 엔진 (4단계 임계값 판정 · FCM/APNs 발송)
      ↓ WSS · Push 알림

[4. 모바일 앱]
  React Native · iOS / Android · 실시간 수위 + 4단계 경보
```

모든 데이터 흐름은 SCADA에서 모바일 방향의 단방향입니다. 외부에서 SCADA로의 제어 명령은 DMZ 단방향 게이트웨이에서 물리적·논리적으로 차단되어야 합니다.

## 개발 일정 체크리스트

### 1단계 · 요구사항 정의 (1~2주)

- [ ] 현재 SCADA 프로토콜 조사 (OPC UA / Modbus / DNP3)
- [ ] 22개 펌프장 센서 현황 및 태그 구조 파악
- [ ] 경보 임계값 4단계 정의 (주의 / 경계 / 심각 / 위험)
- [ ] 30초 주기 허용 지연 · 유실 기준 설정
- [ ] 사용자 역할 정의 (담당자 / 상황실 / 현장 / 관리자)

### 2단계 · 네트워크 · 보안 아키텍처 설계 (1~2주 · 핵심)

- [ ] 대구시 정보보안 부서 사전 협의 및 보안성 검토 신청
- [ ] 단방향 게이트웨이 vs 방화벽 기반 DMZ 방식 결정
- [ ] 데이터 흐름 방향 확정 (SCADA pull vs push)
- [ ] TLS 1.3 + 기기 인증서 + API 토큰 체계 설계
- [ ] 호스팅 결정 (공공클라우드 G-Cloud / 자체 서버실)
- [ ] 아키텍처 다이어그램 문서화 및 검토 승인

### 3단계 · 백엔드 개발 (3~4주)

- [ ] SCADA 데이터 수집기 구현 (OPC UA / Modbus 클라이언트)
- [ ] TimescaleDB 스키마 설계 및 hypertable 구성
- [ ] FastAPI REST 엔드포인트 (현재 / 이력 / 경보 / 메타)
- [ ] WebSocket 실시간 스트리밍 서버
- [ ] 알람 엔진 (4단계 판정 · 히스테리시스 · 채터링 방지)
- [ ] FCM (Android) + APNs (iOS) 푸시 연동
- [ ] 인증 · 인가 (JWT) 및 관리자 API 분리

### 4단계 · 모바일 앱 개발 (3~4주)

- [ ] 메인 화면 · 22개소 지도 + 상태별 색상 표시
- [ ] 펌프장 상세 · 실시간 수위 차트 + 24시간 추이
- [ ] 경보 화면 · 4단계별 알림음 · 진동 · 풀스크린
- [ ] 오프라인 대응 · 마지막 값 캐싱 + 연결 상태 표시
- [ ] 당직자 인수인계 · 야간 교대 경보 책임자 지정
- [ ] 푸시 토큰 등록 + 배터리 최적화 예외 가이드

### 5단계 · 테스트 · 검증 (2~3주 · 필수)

- [ ] 부하 테스트 · 22개소 동시 수집 + 다수 동접
- [ ] 장애 시나리오 · 연결 끊김 · 센서 오류값 튐
- [ ] 오경보 (false positive) 검증 및 히스테리시스 튜닝
- [ ] 파일럿 펌프장 1~2곳 실제 운영 테스트
- [ ] 우기 전 안정화 완료 (6월 이전 목표)

### 6단계 · 단계적 배포 · 운영 (지속)

- [ ] 5개소 확장 배포 (1차 롤아웃)
- [ ] 22개소 전체 배포 완료
- [ ] 관리자용 운영 대시보드 개설
- [ ] 담당자 교육 및 운영 매뉴얼 배포
- [ ] 경보 통계 · 로그 분석 월간 리포트 운영

## 핵심 유의사항

- **2단계(네트워크·보안)와 5단계(오경보 검증)가 전체 일정의 병목**이다. 2단계는 보안 심사 회신이 늦어지면 뒷단이 전부 밀리므로 1단계와 병행 착수를 권장한다.
- **단방향 데이터 흐름이 절대 원칙**이다. 외부 → SCADA 방향의 제어 경로가 있으면 보안 심사를 통과할 수 없다.
- **오경보 한 번이 시스템 신뢰를 무너뜨린다.** 히스테리시스와 지속시간 조건(예: 위험 수위 60초 이상 유지)을 반드시 적용한다.
- **오프라인 상태 표시**가 중요하다. 오래된 수위 값을 현재 값처럼 표시하면 절대 안 된다.
